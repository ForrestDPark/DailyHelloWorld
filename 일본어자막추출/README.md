- 로컬 경로: `/Users/forrestdpark/Desktop/PDG/DailyHelloWorld_/일본어자막추출/whisper_series_stream.sh`
- 원래는 macOS 단축어(Shortcuts)로 실행하던 스크립트를 그대로 저장소에 옮긴 것. 단축어 없이 터미널에서 직접 실행하거나, shift_alarm 메뉴바 앱에서도 실행 가능(아래 참조).

## 사용법

```bash
./whisper_series_stream.sh [영상 폴더 경로]   # 생략하면 현재 폴더
```

폴더 안의 `.mp4`/`.webm`/`.mkv`/`.mov` 파일을 전부 찾아서 순서대로 처리한다. 처리 중인 진행 상황은 새 터미널 창에서 실시간으로 보여준다(스크립트 자신은 그 터미널 창을 띄우고 종료 — 실제 작업은 새 창에서 백그라운드로 계속 진행됨).

## 고음영상 추출 (`extract_high_pitch_video.py`) — 2026-07-23 추가

whisper 자막 파이프라인과는 완전히 독립적인 별도 도구. Notion/메모 앱/EPUB 아무것도 안 건드리고, **영상에서 오디오를 뽑아 음높이(pitch)를 분석해서, 평균보다 확연히 높은 "고음 구간"만 모아 원본 영상에서 잘라 이어붙인 하이라이트 영상**을 만든다.

```bash
# 영상 하나 처리 (기본값: 유성음 pitch 분포 상위 10%를 고음 기준으로, 1초 미만 구간은 버림)
python3 extract_high_pitch_video.py MIAA-444.mp4

# 기준을 더 빡빡하게(상위 5%만) + 짧은 스파이크도 허용 + 최대 10구간만
python3 extract_high_pitch_video.py MIAA-444.mp4 --top-percent 5 --min-duration 0.5 --max-clips 10

# 폴더 안 영상 전부 처리
python3 extract_high_pitch_video.py ./av2/
```

**동작 원리**:
1. `ffmpeg`로 오디오를 wav(모노 22050Hz)로 추출 — `temp_<파일명>_pitch.wav`로 캐시되어 재실행 시 재사용(휘퍼 파이프라인의 `temp_*.wav` 캐시 관례와 동일).
2. `librosa.pyin`(확률적 YIN 알고리즘)으로 프레임마다 기본주파수(pitch, Hz)를 추정 — 유성음 구간에서만 유효한 값이 나옴(무음/자음만 있는 프레임은 제외).
3. 유성음 프레임들의 pitch 분포에서 평균값과, `--top-percent`(기본 10%) 상위 퍼센타일을 "고음 기준"으로 잡는다. **평균 자체가 아니라 분포 상위 퍼센타일을 기준으로 쓰는 이유**: 목소리 톤은 사람/장면마다 기준선이 다 달라서, 평균보다 살짝만 높아도 잡히면 거의 전체가 걸리거나 반대로 거의 안 걸릴 수 있다 — 상위 N%로 잡으면 항상 "그 영상 안에서 상대적으로 가장 높은 구간"이 뽑힌다.
4. 기준을 넘는 프레임들을 `--max-gap`(기본 0.5초) 이내로 붙어있으면 하나의 구간으로 묶고, `--pad`(기본 0.3초)만큼 앞뒤 여유를 준 뒤, `--min-duration`(기본 1초)보다 짧은 구간은 스파이크로 보고 버린다.
5. `ffmpeg`로 각 구간을 원본 영상에서 잘라(재인코딩) `concat`으로 이어붙여 `<파일명>_고음영상.mp4`를 만든다.
6. `bgm/` 폴더에 mp3가 있으면(아래 "고음영상 배경음 입히기" 참고) 그중 하나를 무작위로 골라 배경음으로 믹스한 `<파일명>_고음영상_bgm.mp4`도 추가로 만든다.

**의존성**: `librosa`, `soundfile` — 스크립트 실행 시 없으면 자동으로 `pip install`(anaconda python3 기준, 다른 스크립트들이 pykakasi/edge-tts를 자동 설치하는 것과 동일한 관례). `ffmpeg`는 기존 파이프라인과 공유.

**아직 실제 영상으로 검증 전** — 이 세션은 ffmpeg/librosa를 실행할 수 없는 클라우드 컨테이너라, 구간 병합 로직(`find_high_pitch_segments`)과 배경음 폴더 스캔 로직(`pick_random_bgm`)만 합성 데이터로 단위 테스트했고 전체 파이프라인(오디오 추출 → 실제 pitch 분석 → 영상 자르기 → 배경음 믹스)은 맥에서 직접 돌려서 확인 필요. 처음 돌려보고 결과가 기대와 다르면(너무 많이/적게 잡힘) `--top-percent`, `--min-duration`, `--max-gap` 값을 조정해서 재실행하면 됨(오디오 캐시가 있어서 pitch 분석부터는 빠르게 재시도 가능).

### 고음영상 배경음 입히기

`일본어자막추출/bgm/` 폴더에 mp3 파일을 넣어두면, `extract_high_pitch_video.py`가 고음영상을 만든 뒤 자동으로 그중 하나를 무작위로 골라 배경음으로 얹은 `<파일명>_고음영상_bgm.mp4`를 추가로 생성한다. 원본 대사 오디오는 그대로 유지하고 그 위에 낮은 볼륨으로 mp3를 섞는 방식(ffmpeg `amix` 필터) — mp3가 영상보다 짧으면 자동으로 반복 재생해서 영상 길이만큼 채우고, 영상 길이에 맞춰 잘린다.

```bash
# bgm/ 폴더에 mp3가 있으면 기본 동작으로 자동 믹스됨 (별도 옵션 불필요)
python3 extract_high_pitch_video.py MIAA-444.mp4

# 배경음 볼륨 조절 (기본 0.25 = 25%)
python3 extract_high_pitch_video.py MIAA-444.mp4 --bgm-volume 0.15

# bgm/ 대신 다른 폴더의 mp3 사용
python3 extract_high_pitch_video.py MIAA-444.mp4 --bgm-dir /path/to/other_bgm

# 배경음 없이 고음영상만
python3 extract_high_pitch_video.py MIAA-444.mp4 --no-bgm
```

`bgm/` 폴더가 비어있으면(아직 mp3를 안 올렸으면) 자동으로 건너뛰고 안내 메시지만 출력 — 에러 아님. mp3를 커밋해서 `bgm/`에 넣기만 하면 바로 다음 실행부터 적용됨.

## 처리 흐름

1. 영상 길이가 45분(2700초) 이상이면 2편으로 분할, 아니면 통으로 처리.
2. 각 파트: ffmpeg로 오디오 추출(16kHz mono, 대역폭 필터+노멀라이즈) → whisper-cli로 일본어 자막(.srt) 생성 → 파이썬 워커가 3문장씩 묶어서:
   - 각 문장 시점의 영상 프레임을 캡처해 3장을 가로로 합친 썸네일 생성(4MB 넘으면 자동 압축, 생성 후 `sips`로 JFIF 정규화 — 2026-07-24, 아래 "검은 화면" 항목 참조)
   - 구글 번역 비공식 엔드포인트로 한국어 번역
   - `pykakasi`로 한자에 후리가나 병기 (예: `漢字(かんじ)`)
   - **대사 3줄을 통째로 edge-tts(`ja-JP-NanamiNeural`, -10% 속도)에 넘겨 mp3로 합성** (2026-07-24 — 예전엔 영상에서 그 구간을 잘라 이어붙였는데, 끊김/공백이 많다는 피드백으로 TTS 합성으로 교체. 아래 "대사 오디오를 TTS로 교체" 항목 참조)
   - 노션 데이터베이스에 새 페이지(영상당 1개, 2편이면 편당 1개) 생성 후 이미지+원문/번역 블록을 계속 append
   - 애플 메모 "LanguageStudy" 폴더에 같은 제목으로 노트 생성, 원문 텍스트 누적 기록
   - 로컬 `.md` 파일에 이미지+TTS 오디오(mp3)+원문/번역을 계속 append (EPUB 재료)
3. 파트별 자막을 시간축 보정해서 영상 전체의 통합 `.srt` 생성.
4. 쌓인 `.md` 파일(들)을 pandoc으로 EPUB 변환. 표지는 영상에서 새로 캡처한 세로 이미지에 제목을 입혀서 생성, 스타일은 `epub_style.css`(일본어 크고 금색, 한국어 작고 회색). 8세트(대사 24줄)마다 "장면 N" 소제목으로 나눠서 EPUB 목차에서 바로 이동 가능(2026-07-24 개편, 아래 참조).
5. 완성된 EPUB을 Obsidian Study 폴더에 자동 복사(있으면).
6. 오디오 캐시(`temp_*.wav`)와 자막 캐시(`temp_*.wav.srt`)는 재실행 시 재사용됨 — 지우면 처음부터 다시 처리.

## 알려진 문제 / 수정 이력

### ★★ EPUB 생성 실패 — `--epub-version=3` 옵션이 pandoc 3.x에서 삭제됨 (2026-07-23)

**증상**: 로그에 `⚠️ EPUB 생성 실패`만 뜨고 이유를 알 수 없음.

**원인**: 원본 스크립트가 `pandoc ... --epub-version=3 ... 2>/dev/null`로 실행했는데, `--epub-version` 옵션이 pandoc 3.x(사용자 환경: 3.9.0.2)에서 완전히 제거됐다. `pandoc: Unknown option --epub-version.`로 즉시 실패하는데, 에러 출력을 `2>/dev/null`로 버려서 진짜 원인이 안 보였다.

**수정**:
1. `--epub-version=3` 옵션 삭제. `-o *.epub`만으로 이미 EPUB3로 생성되므로 버전 지정이 애초에 불필요.
2. `2>/dev/null` 대신 `${MYTMP}/pandoc_<파일명>.log`로 stderr를 저장, 실패 시 그 로그를 바로 화면에 출력하도록 변경 — 앞으로 비슷한 문제가 생겨도 원인이 바로 보임.

재현/검증: 실패했던 `MIAA-444` 작업 폴더(`~/Desktop/BlogImage/AV/`)에서 `--epub-version=3` 뺀 명령으로 직접 재실행해서 정상적으로 46MB EPUB이 만들어지는 것 확인함(표지 이미지 관련 `Could not determine image type` 경고는 뜨지만 무해함 — 실제로는 유효한 JPEG이고 pandoc의 크기 감지 로직만 못 읽는 것, 1440x270처럼 세로로 아주 납작한 비율이라 그런 것으로 추정).

### iTerm 자동화(Automation) 권한 문제 → `.command` + `open -a iTerm` 방식으로 교체 (2026-07-23, 2026-07-24 iTerm으로 재확정)

원본 스크립트는 맨 끝에서 `tell application "iTerm" ... create window ...`로 새 창을 열었는데, 이건 macOS 자동화 권한이 필요하다. 사용자가 직접 터미널에서 실행할 땐 문제없지만, **shift_alarm 메뉴바 앱(launchd 백그라운드 프로세스)에서 호출하면 권한 팝업 자체가 안 떠서 조용히 실패한다** — shift_alarm 프로젝트의 이북리더에서 이미 겪은 것과 동일한 문제(그 프로젝트 README 8-1 참조). 그래서 실행 가능한 `.command` 파일을 만들고 `open`으로 여는 방식으로 바꿨다 — 권한이 전혀 필요 없다.

**★ 2026-07-24 정정**: 처음엔 `open -a Terminal`로 바꿨었는데, 그러면 장면 미리보기(`imgcat`)가 안 보인다는 걸 사용자가 지적함 — `imgcat`은 iTerm2 전용 인라인 이미지 이스케이프 시퀀스라 일반 Terminal.app은 못 그린다. 확인해보니 **`open -a iTerm <파일>.command`도 Terminal과 동일하게 자동화 권한 없이 스크립트를 실행해준다** — 애초에 Automation(`kTCCServiceAppleEvents`) 권한이 필요한 건 `tell application "X"` 같은 AppleEvent를 보낼 때뿐이고, `open`은 Finder에서 파일을 더블클릭하는 것과 같은 취급(Launch Services)이라 대상 앱이 Terminal이든 iTerm이든 상관없이 권한이 필요 없다. 그래서 최종적으로 `open -a iTerm`으로 되돌렸다 — 권한 문제도 없고 imgcat 미리보기도 정상 동작.

### ★★ EPUB 가독성 전면 개편 (2026-07-24)

사용자 피드백: 표지가 이상함 / 전체적으로 가독성이 없음 / 페이지당 글자 수가 너무 적음 / 1편·2편 안에서도 목차 분할이 있었으면 함. 원인과 수정 내역:

1. **표지가 깨져 보이던 원인**: EPUB 안 `cover.xhtml`을 까보면 `<image width="0" height="0">`로 박혀있었다 — ffmpeg가 만든 jpg가 표준 JFIF(APP0) 헤더 없이 저장되는 경우가 있어서 pandoc의 이미지 크기 판독기가 가로/세로를 못 읽고 0으로 처리해버린 것(`file` 명령으로는 정상 JPEG로 보여서 헷갈리기 쉬움 — pandoc이 쓰는 판독기가 더 엄격함). `sips -s format jpeg cover.jpg --out cover.jpg`로 다시 저장하면 표준 JFIF로 정규화되어 해결된다 — EPUB 재빌드 후 `cover.xhtml`에 정확한 `960x1440`이 찍히는 것 확인함.
2. **표지 이미지 자체도 교체**: 예전엔 본문에 쓰던 3장 가로 합성 썸네일(1440x270, 배너처럼 납작함)을 표지로 재사용했다. 지금은 영상에서 별도로 세로 비율(960x1440, 2:3)로 새로 캡처하고, 하단에 반투명 검은 박스 + 제목 텍스트(영상 파일명)를 `drawtext`로 입혀서 표지답게 만든다. 캡처 시점은 영상 길이의 4%(최소 5초, 최대 90초) 지점 — 도입부일 가능성이 높은 이른 시점을 고른 것.
3. **페이지당 글자 수가 적었던 원인**: 장면 이미지가 `width:100%`로 각 세트(대사 3줄)마다 화면 전체 폭을 차지해서, 화면(페이지) 한 장에 텍스트가 몇 줄 안 들어갔다. 이미지를 `38%` 폭으로 축소하고 오디오 플레이어도 `60%`로 줄여서, 같은 화면에 훨씬 많은 대사 줄이 들어가게 했다.
4. **1편/2편 안에서 목차 분할**: 8세트(대사 24줄)마다 "🎬 장면 N" 소제목을 넣어서 EPUB 목차(TOC)에서 장면 단위로 바로 이동할 수 있게 했다. **★ 함정**: 처음엔 raw HTML로 `<h2 class="scene">...</h2>`를 그냥 썼는데, pandoc의 `--toc` 목차 생성기는 마크다운 문법으로 파싱된 `Header` 노드만 인식하고 raw HTML 블록은 그냥 불투명하게 통과시켜버려서 목차에 전혀 안 잡혔다. 마크다운 네이티브 헤더 문법 `## 제목 {.scene}`(pandoc의 헤더 속성 확장 — `{.클래스}`로 CSS 클래스도 그대로 부여됨)으로 바꾸니 목차에 정상적으로 잡힘. `--toc-depth=2`도 추가해서 파트(h1)뿐 아니라 장면(h2)까지 목차에 나오게 함.
5. **개요 페이지 추가**: 각 파트 md 맨 앞(표지/제목 페이지 바로 다음)에 그 파트의 대사 문장 수·장면 수·표기 방식을 요약한 박스를 넣었다. ★ 대사 내용 자체를 AI로 요약하는 기능은 일부러 넣지 않았다 — 성인 영상 대사를 자동 요약하는 파이프라인을 만드는 것은 별개의 판단이 필요한 문제라 통계 기반 개요로 대신함.
6. **검증**: 이미 처리 완료된 `MIAA-444`, `435MFCS-058` 작업 폴더의 기존 md/이미지/오디오를 그대로 활용해(재번역 없이 텍스트만 재구성) 새 스타일로 EPUB을 다시 빌드해서 표지 치수(960x1440 정상)와 목차 중첩(파트 밑에 장면들이 하위 항목으로 들어가는 것) 둘 다 확인함.

### ★★ 본문 중간중간 이미지가 검은 화면/빈 박스로 나오는 문제 (2026-07-24)

표지에서 겪은 것과 **완전히 같은 원인**이 본문 장면 썸네일에도 있었다: `merge_images()`가 ffmpeg로 만드는 jpg 중 일부가 표준 JFIF(APP0) 헤더 없이 저장되고, 그런 파일은 e리더가 가로/세로를 못 읽어서 빈 박스(까맣게)로 렌더링한다. 실제로 확인해보니 **영상 하나 안에서도 세트(장면)에 따라 갈렸다** — `MIAA-444`는 183개 썸네일 전부 헤더가 없었고, `435MFCS-058`은 288개 전부 정상이었다. 정확한 ffmpeg 내부 분기(단일 프레임 `scale`만 쓴 경우 vs 3장 `hstack` 합성한 경우 등)까지 완전히 특정하지는 못했지만, 원인 특정보다 결과를 항상 정규화하는 쪽이 더 확실하고 비용도 낮아서 **`merge_images()`가 만든 파일을 매번 `sips -s format jpeg`로 다시 저장**하도록 고쳤다 — 표지에 이미 쓰던 처리를 세트 썸네일에도 그대로 적용한 것. 기존에 이미 만들어둔 `MIAA-444`/`435MFCS-058`/`AV3` 폴더의 7개 작업물도 이미지 전체를 일괄 `sips` 정규화하고 EPUB을 다시 빌드해서 반영함.

### ★★ 대사 오디오를 "영상에서 자른 실음성" → TTS 합성으로 교체 (2026-07-24)

**문제**: 영상에서 대사 구간(`c['start']`~`c['end']`, 앞뒤 여유 0.15~0.3초)을 잘라 세트별로 이어붙이던 방식이 실제로 들어보면 문장 중간에 끊기거나 빈 공백이 많았다 — whisper가 잡은 타임스탬프가 항상 정확하진 않고, 문장 사이 간격을 그대로 이어붙이다 보니 자연스럽지 않았다.

**해결**: 세트(대사 3줄)를 하나의 문자열로 합쳐서 `edge-tts`(`ja-JP-NanamiNeural`, `-10%` 속도 — `ebook_reader.py`가 영어 TTS에 쓰는 것과 같은 라이브러리)에 통째로 넘겨 mp3를 직접 합성한다. 이러면서:
- 영상에서 개별 문장 오디오를 자르는 ffmpeg 호출, 그걸 이어붙이는 `concat_audio_clips()`, wav→mp3 변환까지 세 단계가 통째로 없어지고 TTS 합성 한 번으로 끝남(코드도 단순해짐).
- 문장 하나하나가 아니라 **세트 단위로 한 번에 합성**하는 이유: 문장별로 3번 따로 합성해서 이어붙이면 그 사이에 어색한 침묵이 생기고, 네트워크 호출도 3배로 늘어난다. 문장을 이어붙인 텍스트를 한 번에 합성하면 자연스러운 억양으로 이어진다.
- **트레이드오프**: 실제 배우 음성이 아니라 TTS 음성으로 완전히 대체된다. 실제 음성도 같이 듣고 싶다면(둘 다 넣으면 세트당 오디오 플레이어가 2개가 되어 페이지 밀도가 다시 나빠짐) 토글 방식으로 확장 가능 — 필요하면 요청할 것.
- 기존에 이미 만들어둔 작업물(`MIAA-444`, `435MFCS-058`, `AV3`의 7개)은 이번에 TTS로 재생성하지 않았다(수백~수천 건의 TTS 호출이 필요해 시간이 많이 걸림) — 실제 음성 오디오를 그대로 유지한 채 검은 화면 버그만 고쳐서 다시 빌드했다. **새로 처리하는 영상부터 TTS가 자동 적용된다.**

### ★ av2 폴더 재작업 — TTS 파이프라인 실전 검증 + 표지 생성 실패 (2026-07-24)

`av2` 폴더(`MKMP-533`, `MUDR-393`)는 급하게 만든 임시 변환 스크립트로 옛 md를 재구성했다가 실패했다 — EPUB을 열면 페이지가 5~6개밖에 없고 내용도 거의 안 보였다(정확한 원인은 못 밝혔지만, 임시 스크립트가 옛 형식을 완전히 정확하게 파싱하지 못했던 것으로 추정). **그래서 임시 변환 대신 실제 파이프라인 스크립트를 처음부터 다시 돌렸다** — whisper 자막(`.wav.srt`)과 장면 캡처는 캐시가 있어서 재사용되고, 번역+TTS+노션+메모+md 생성만 새로 진행됨.

이 과정에서 새 버그 하나 발견: **표지 생성 ffmpeg 명령이 실전 파이프라인 안에서는 조용히 실패**했다(터미널에서 똑같은 명령을 직접 실행하면 항상 성공하는데, 오래 도는 백그라운드 실행 중에는 실패 — 정확한 원인 미확정, 리소스 경합으로 추정). 표지 없이 EPUB이 만들어졌길래 수동으로 표지만 다시 만들어 끼워넣었다. 재발해도 바로 원인을 알 수 있게 `-loglevel error`로 출력을 버리던 걸 로그 파일로 남기도록 고쳤다(실패 시 `⚠️ 표지 캡처 실패 — 로그: ...` 뒤에 마지막 20줄을 바로 출력).

**검증 결과**: `MKMP-533`(76세트, 76개 TTS mp3) / `MUDR-393`(119세트, 119개 TTS mp3) 둘 다 표지(960x1440)·본문 세트 수·오디오 파일 수가 정확히 일치하는 것까지 확인함 — 실제 파이프라인으로 처음부터 다시 도는 것 자체는 정상 동작 확인됨(표지 생성만 예외적으로 방어 로직 추가).
