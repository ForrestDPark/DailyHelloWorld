# 카톡 대화 정리 파이프라인

이 문서는 **새 세션(다른 기기·다른 클라이언트 포함)에서도 이전 대화 맥락 없이 바로 같은 작업을 이어할 수 있도록** 지금까지 확정된 절차를 전부 기록해둔 것이다. 사용자가 "OO방 정리해줘" 또는 "다음 방 정리해줘"라고 말하면 아래 내용만 보고 그대로 실행하면 된다.

## 0. 트리거 & 개요

- 트리거: 사용자가 "OO방 정리해줘" / "다음 방으로 넘어가자" 라고 말하면, **그 방 하나만** 아래 1~3번 절차로 정리한다(신규 방 최초 정리). 이미 정리한 방을 다시 정리하는 경우(주 1회 반복, 사용자가 "카톡방 정리" 정기 요청 시)는 5번 섹션 "주간 증분 업데이트" 절차를 따른다 — 전체를 다시 쓰지 않고 마지막 정리 이후 내용만 이어붙인다.
- 원본 데이터: 아이폰 카카오톡에서 "대화 내보내기"로 받은 zip 파일들. `카톡정리/원본대화/*.zip`에 보관되어 있다(맥에서 git push로 올리거나, github.com 웹 업로드로 추가 가능 — 25MB 넘는 파일은 git CLI/Desktop 필요).
- 정리 결과를 쓰는 곳: 노션 **인덱스 페이지 "카톡정리"** — `https://app.notion.com/p/3a632a1eae80803e918de11510d4c66a`. **★ (사용자 명시 요청) 방마다 독립된 하위 페이지를 하나씩 만든다.** 인덱스 페이지에는 번호가 매겨진 목록만 있고, 정리가 끝난 방은 그 번호가 하위 페이지로의 `<page url="...">` 링크가 되고, 아직 정리 안 한 방은 `방이름 — *(다음 정리 예정)*`으로 표시된다.
- 진행 현황: 1~3번(AIMAX, AI자동화마케팅(개발의신), [데이터팝콘] Agentic AI 자동화 커뮤니티)까지 완료(전체 흐름 요약 + Q&A, 독립 페이지로 분리됨). 나머지는 미완료.

## 1. 데이터 추출 + 파싱

1. 해당 방의 zip을 임시 폴더에 압축 해제한다 (`unzip -o "카톡정리/원본대화/<파일명>.zip" -d /tmp/kakao_extract/<방이름>`). zip 안에는 `Talk_*.txt`(대화 원본)와 사진 파일 몇 장이 들어있다.
2. `카톡정리/parse_kakao.py`로 파싱한다:
   ```bash
   python3 카톡정리/parse_kakao.py "/tmp/kakao_extract/<방이름>/Talk_*.txt" \
     --outdir /tmp/kakao_extract/<방이름> \
     --admin-names "운영진으로_보이는_이름1" "이름2"
   ```
   출력:
   - `messages.json` — 입퇴장/오픈채팅봇 공지를 제외한 전체 메시지 `{date, time, sender, text}` 리스트.
   - `admin_posts.json` — `--admin-names`로 지정한 발신자의 장문(60자 이상) 메시지만. 운영진이 누구인지는 처음 몇백 줄을 훑어보고 판단한다(공지·홍보를 반복해서 올리는 사람).
   - `qa_candidates.json` — `?`나 의문형 종결어미(~나요/~까요 등)로 끝나는 메시지 + 그 뒤 맥락(기본 20줄). **이 파일은 그대로 쓰지 말고**, 아래처럼 텍스트만 뽑은 축약 리스트를 먼저 만들어서 훑어본다(파일이 너무 커서 통째로 읽으면 컨텍스트를 낭비함):
   ```bash
   python3 -c "
   import json
   with open('/tmp/kakao_extract/<방이름>/qa_candidates.json', encoding='utf-8') as f:
       cands = json.load(f)
   for i, c in enumerate(cands):
       q = c['question']
       print(f'{i}\t{q[\"date\"]}\t{q[\"sender\"]}\t{q[\"text\"][:100].replace(chr(10),\" \")}')" > /tmp/kakao_extract/<방이름>/qa_list.tsv
   ```
   이 tsv를 Read 도구로 읽어서(한 번에 25000토큰 넘으면 offset/limit로 나눠 읽기) 질문 후보를 훑는다.

## 2. 정리 산출물 — 방마다 아래 섹션들을 만든다

노션 하위 페이지에 `*(다음 정리 예정)*` 대신 아래 섹션들을 순서대로 채운다. 필수는 전체 흐름(2-1)과 Q&A(2-2)이고, 나머지(2-3\~2-9)는 해당 내용이 있을 때만 만든다(없으면 억지로 만들지 않고 생략하거나 "확인되지 않음"으로 정직하게 표시).

### 2-1. 전체 흐름 요약 (`### 전체 흐름`)

- 날짜·주제별로 무슨 대화가 오갔는지 **서사식으로** 정리한다. 잡담·분위기도 포함(사용자가 "전체 흐름 요약, 잡담 포함"으로 명시 확정).
- 메시지를 전부 한 줄씩 읽지 말고, 다음 순서로 효율적으로 파악한다:
  1. `messages.json`에서 발신자별 메시지 수를 세서(`Counter`) 누가 운영진/헤비유저인지 파악.
  2. `admin_posts.json`을 날짜순으로 훑어 공지·이벤트·캠페인의 타임라인(뼈대)을 잡는다.
  3. 원본 txt 앞부분(500~1000줄 정도)을 Read로 직접 읽어 그 방 특유의 분위기·말투·사건(바이럴 사건 같은 것)을 파악한다.
  4. 나머지는 `admin_posts.json`과 `qa_list.tsv`로 훑으며 흐름을 보완한다.
- 형식: 기간·메시지 수·방 성격 요약 1~2문장 → 시기별 불릿 리스트(세팅기/특이사건/반복패턴 등 자연스러운 구간으로 나눔) → 한 줄 요약.

### 2-2. Q&A 토글 (`### Q&A (중복 질문 병합, 여러 명 답변 통합)`)

**★ (사용자 명시 요청) 질문·답변이 오간 부분은 Q&A 형식으로 토글(접었다 펼치는 블록)로 만든다.** 규칙:

1. `qa_list.tsv`를 훑으며 **진짜 정보 요청 질문**만 고른다. 걸러야 할 것:
   - 잡담·감탄사성 물음표(`"이거 전략인가요? ㅋ"`, `"오옹?"` 같은 리액션)
   - 수사의문문(대답을 바라지 않는 질문)
   - 광고/공지 메시지 안에 우연히 물음표가 들어간 경우
2. **같은 취지의 질문이 여러 번(다른 날짜, 다른 사람) 나오면 하나로 병합**한다 — 예: "환불 어떻게 하나요", "결제했는데 확인이 안 돼요", "무통장입금 오류나요" → "결제/환불 문의" 하나로.
3. **답변이 여러 사람(운영진 + 회원들)에게서 나왔다면 하나의 정합적인 답으로 통합**해서 쓴다. 서로 다른 의견이 갈리는 주제(예: 어떤 AI 툴이 좋은가)는 "의견이 갈린다"는 사실 자체를 답으로 정리하고 각 의견을 짧게 병기한다.
4. 대화에서 **명확한 답이 끝내 나오지 않은 질문도 그대로 신든다** — 답 대신 "대화 기록상 확인되지 않음 / 미해결로 표시"라고 정직하게 쓴다(답을 지어내지 않는다).
5. 노션 토글 마크다운 문법(이 워크스페이스에서 실제 토글로 렌더링되는 것을 확인함):
   ```
   <details>
   <summary>Q. 질문(병합된 경우 대표 질문 문구로)</summary>
   답변 내용(1~3문장, 통합된 답)
   </details>
   ```
   여러 개를 연달아 쓰면 각각 별도 토글이 된다.
6. **★ 마크다운 함정 주의**: 괄호 안에 URL을 맨텍스트로 쓰면(`채널(pf.kakao.com/_xxx)로 문의`) 노션이 괄호 뒤 텍스트까지 링크로 잘못 묶어서 깨진다(`[pf.kakao.com/_xxx)로](...)로)` 같은 중복 텍스트 발생, 실제로 이번 세션에서 겪음). URL은 반드시 `[표시텍스트](URL)` 형식으로 명시적으로 감싸거나, 괄호 밖으로 빼서 "채널 주소: pf.kakao.com/_xxx" 식으로 쓴다.

### 2-3. 이미지 (있는 경우만) (`### 참고 이미지`)

**★ (사용자 명시 요청) 그 방 zip 안에 이미지가 있으면, 사이즈를 작게 줄여서 노션 페이지에 올린다.**

1. `/tmp/kakao_extract/<방이름>/` 압축 해제 결과에 `.jpg`/`.png`/`.jpeg` 등 첨부 이미지가 있는지 확인한다. 없으면 이 섹션 자체를 생략한다(방마다 이미지가 없을 수 있음 — 예: room 2는 이미지 없음).
2. **이미지가 있으면 올리기 전에 Read 도구로 하나씩 실제로 열어본다.** 그리고 아래 기준으로 제3자 개인정보(PII) 스크리닝을 한다 — 실명, 이메일, SNS 계정, 포트폴리오 URL, 신원이 드러나는 얼굴 사진 등이 보이면 **그 이미지는 제외**한다. 제품 화면·UI 캡처·일반 홍보 이미지처럼 특정 개인을 식별할 수 없는 것만 올린다.
3. 올리기로 한 이미지는 PIL로 가로 500px 정도로 축소한다:
   ```python
   from PIL import Image
   im = Image.open(src)
   w, h = im.size
   im.resize((500, int(h * 500 / w)), Image.LANCZOS).save(dst, optimize=True)
   ```
4. 축소한 이미지를 `카톡정리/images/<방이름>/`에 저장하고 `claude/session-5kpo7t`와 `main` 양쪽에 커밋+푸시한다.
5. 노션 페이지에는 GitHub raw URL로 임베드한다: `https://raw.githubusercontent.com/ForrestDPark/DailyHelloWorld/main/%EC%B9%B4%ED%86%A1%EC%A0%95%EB%A6%AC/images/<방이름(퍼센트인코딩)>/<파일명>` — `카톡정리` 한글 경로의 퍼센트 인코딩은 `%EC%B9%B4%ED%86%A1%EC%A0%95%EB%A6%AC`.
6. PII 때문에 제외한 이미지가 있으면, 그 사실을 페이지에 짧게 밝힌다(예: "본인 식별 정보가 포함된 이미지 N장은 제외함"). 몰래 빼지 말고 투명하게 남길 것.

### 2-4. 공유된 링크 (`### 공유된 링크`)

**★ (사용자 명시 요청) 대화에서 공유된 링크가 있으면 리스트로 정리한다.**

1. `messages.json`에서 정규식(`https?://\S+`)으로 URL을 전부 추출하고 URL 기준으로 중복 제거한다.
2. 유형별로 묶어서 정리한다 — 예: 공식 카페/공지, 오픈채팅방 입장 링크, 무료강의·라이브 다시보기, 자체 툴/플랫폼 사이트, 문의(CS) 채널, 회원 개인이 참고·홍보용으로 공유한 링크(개인 블로그/SNS/유튜브 등).
3. **링크 수가 너무 많으면(대략 50개 초과) 개별 나열하지 말고 도메인별 건수 + 대표 링크 몇 개만 소개하는 요약 방식으로 전환한다** — 방마다 링크 밀도 차이가 커서(예: 어떤 방은 30개, 어떤 방은 300개 이상), 후자의 경우 전부 나열하면 페이지가 읽기 힘들어진다.
4. 회원이 자기 SNS/블로그 링크를 홍보 목적으로 스스로 공유한 경우는 이미 본인이 공개한 콘텐츠이므로 그대로 링크를 남겨도 된다 — 단, 2-3의 이미지 PII 스크리닝과 같은 기준으로 봤을 때 명백히 제3자를 특정하거나 사생활을 침해하는 링크라면 제외한다.

### 2-5. 토론 요약 (`### 토론 요약`)

**★ (사용자 명시 요청) 회원들 사이에 실제로 의견이 오간 토론(논쟁)이 있으면 간단히 요약해서 볼 수 있게 만든다.** Q&A(운영진 답변 중심)와는 다르게, **여러 회원이 서로 다른 의견을 주고받은 스레드**를 찾는 것이 핵심이다.

1. 찾는 방법: `messages.json`에서 같은 시간대에 서로 다른 발신자 여러 명이 연속으로 등장하는 구간(클러스터)을 추리고, 그 안에서 특정 주제에 대해 서로 다른 입장(찬성/반대, 경험담 vs 반박 등)을 주고받는지 직접 읽어서 판단한다. 단순히 여러 명이 각자 다른 질문을 쏟아내는 구간(=Q&A 후보)과 혼동하지 말 것.
2. Q&A 토글과 같은 `<details><summary>토론. ...` 형식으로 정리하되, 누가 어떤 입장이었는지, 결론이 났는지 안 났는지까지 정직하게 담는다(의견 불일치가 해소되지 않았으면 "합의되지 않았다"고 그대로 쓴다).
3. 뚜렷한 토론 스레드가 없는 방도 있다(운영진 중심 홍보성 방 등) — 이 경우 억지로 만들어내지 말고 "뚜렷한 토론 스레드는 확인되지 않음"이라고 정직하게 쓴다.

### 2-6. 새로운 용어 설명 (`### 새로운 용어 설명`)

**★ (사용자 명시 요청) 대화에 나오는 생소한 용어나 유용한 기술·지식은 구체적으로 설명한다.**

1. 그 방에서 반복적으로 등장하지만 일반적으로 낯선 용어(줄임말, 업계 은어, 그 커뮤니티만의 용어, 특정 툴/서비스 이름 등)를 추린다.
2. 대화 맥락에서 실제로 설명되거나 사용된 방식을 근거로 정의를 쓴다 — 정확한 의미가 대화 기록만으로 불확실하면 "~로 추정됨" 또는 "대화 기록상 자세히 설명되지 않음"이라고 정직하게 표시한다(용어 설명에도 Q&A와 동일하게 답을 지어내지 않는 원칙을 적용한다).
3. 단순 사전적 정의로 끝내지 말고, 그 방 안에서 실제로 어떻게 쓰이는지/왜 중요한지(리스크, 효용 등)까지 구체적으로 설명한다.

### 2-7. 주요 참여자 유형 (`### 주요 참여자 유형`)

**★ (사용자 명시 요청) 자주 대화하는 사람들의 성향과 전문 지식을 유형별로 분류해서 정리한다.**

1. `messages.json`에서 발신자별 메시지 수 상위권(대략 상위 15\\~20명)을 추리고, 각자의 메시지 샘플을 몇 개씩 읽어 대화 패턴(질문형/설명형/실적공유형/감정표현형/경고형 등)과 전문 분야(닉네임에 직업이 붙어 있으면 그것도 활용)를 파악한다.
2. "유형 — 대표 닉네임: 설명" 형식의 불릿 리스트로 정리한다. 한 유형에 여러 명이 묶일 수도 있다(예: 신규/초보 질문형).
3. 닉네임은 카카오톡 채팅 닉네임이며 실명이 아닐 수 있다는 점을 페이지에 명시해서, 실제 신원 특정처럼 오해되지 않게 한다.

### 2-8. 실무 팁 / 활용법 (`### 실무 팁 / 활용법`)

**★ (사용자 명시 요청) "OO 활용법", "OO 발급 조건"처럼 실무적으로 유용한 정보는 전체 흐름 요약에 한 줄로 뭉개지 말고, 별도 섹션에서 구체적으로 설명한다.**

1. 전체 흐름 요약을 쓰다가 "~활용법", "~조건", "~하는 법" 같은 실무 팁이 나오면, 그 팁을 이 섹션의 토글 항목으로 따로 뺀다.
2. 각 항목에는 **대화에서 실제로 오간 메시지 원문(또는 그에 가까운 인용)**을 근거로 무슨 내용이었는지 구체적으로 쓴다 — "~에 대한 활용법을 안내함" 같은 뭉뚱그린 문장으로 끝내지 않는다.
3. **사실관계를 검증할 수 있는 내용(공식 서비스의 정책·기능 등)은 WebSearch로 검증**하고, 검증 결과(일치/불일치/확인불가)를 명시한 뒤 **참고할 수 있는 외부 링크(Sources)를 반드시 첨부**한다. 그 방/운영자만 아는 비공개 정보(자체 브랜드 툴 이름 등)라 웹 검색으로 확인이 안 되면 "미검증으로 표시"라고 정직하게 쓴다(Q&A와 동일한 원칙 — 답을 지어내지 않는다).
4. 형식은 Q&A와 동일한 토글로 통일한다: `<details><summary>팁. ...</summary>` 안에 원문 인용 + 설명 + 웹 검증 결과 + Sources 링크.

### 2-8b. 긴 리스트형 섹션은 토글화 + 실제 발언 인용

**★ (사용자 명시 요청) `### 새로운 용어 설명`, `### 공유된 링크`, `### 주요 참여자 유형`처럼 항목이 많이 쌓이는 섹션은, 평범한 불릿 리스트로 쭉 나열하지 말고 항목별로 `<details><summary>...` 토글로 감싼다.** (Q&A·토론·실무 팁은 이미 토글 형식이라 그대로 유지.)

1. 용어 설명 → `<details><summary>용어. 용어명</summary>` 토글 하나당 용어 하나.
2. 공유된 링크 → 카테고리(공식 공지/무료강의/툴 사이트 등)별로 토글 하나씩, 그 안에 불릿으로 링크 나열.
3. 주요 참여자 유형 → 유형 하나당 토글 하나. **★ (사용자 명시 요청) 각 유형 토글 안에는 그 사람이 실제로 쓴 메시지 원문을 "실제 발언 예시: (날짜) "..."" 형식으로 1~4개 인용해서, 설명이 근거 없는 인상 비평이 아니라 실제 대화에 기반했다는 걸 바로 확인할 수 있게 한다.** `messages.json`에서 해당 발신자로 필터링해 대표성 있는 문장(너무 짧은 리액션 제외)을 골라 그대로 인용한다.
4. 노션에는 페이지 내부 특정 섹션으로 점프하는 앵커 링크 문법이 없으므로(2-9 참고), "그 사람이 쓴 글 목록을 보고 싶다"는 요청은 별도 링크를 만드는 대신 **참여자 유형 토글 안에 발언 예시를 직접 넣는 방식으로 해결**한다 — 같은 토글 안에 있으니 펼치기만 하면 바로 보인다.

### 2-9. 페이지 내 이동 링크 (전체 흐름 ↔ 세부 섹션)

**★ (사용자 명시 요청) 전체 흐름 요약에서 "~기능을 안내함"처럼 다른 섹션(Q&A, 실무 팁 등)에서 더 자세히 다루는 내용을 짧게만 언급하고 넘어가는 경우, 독자가 바로 이동해서 확인할 수 있게 한다.**

1. Notion 마크다운은 페이지 안의 특정 헤딩/토글로 바로 점프하는 앵커 링크 문법을 지원하지 않는다(`notion://docs/enhanced-markdown-spec` 확인 결과, `[텍스트](#헤딩)` 같은 문법은 없음). 대신 아래 두 가지 방법을 함께 쓴다:
   - 페이지 맨 위, "마지막 정리 기준일" 인용구 바로 아래에 `<table_of_contents/>` 블록을 하나 넣는다 — Notion이 모든 헤딩(`### 전체 흐름`, `### Q&A`, `### 실무 팁 / 활용법` 등)으로 바로 이동할 수 있는 클릭형 목차를 자동 생성해준다.
   - 전체 흐름 요약 본문 중, 다른 섹션에서 더 자세히 다루는 내용이 나오면 그 자리에 `(상세: 아래 Q&A "..." 항목 참고)`처럼 어느 섹션의 어느 항목인지 구체적으로 짚어주는 괄호 문구를 붙인다. 목차로 그 섹션까지 이동한 뒤 눈으로 항목을 찾을 수 있도록, 항목 이름을 최대한 정확히 인용한다.
2. 이 방식은 실제 블록 단위 하이퍼링크는 아니지만(Notion API가 지원하는 범위 내 최선), 목차 클릭 한 번 + 정확한 항목명 힌트로 충분히 "바로 이동해서 확인" 가능한 수준을 만든다.

## 3. 노션에 쓰기 (★ 방마다 독립 페이지)

**★ (사용자 명시 요청, 2026-07-23 확정) 방마다 별도의 노션 페이지를 만든다.** 인덱스 페이지 안에 전부 몰아쓰지 않는다.

1. **하위 페이지 생성**: `notion-create-pages`로 인덱스 페이지(`3a632a1eae80803e918de11510d4c66a`)의 자식 페이지를 만든다. `parent: {"type": "page_id", "page_id": "3a632a1eae80803e918de11510d4c66a"}`. 제목은 `"N. 방이름"` 형식(인덱스와 번호를 맞춤), 아이콘은 💬로 통일. `content`의 맨 앞에 `> 📅 **마지막 정리 기준일: YYYY-MM-DD** (대화 내용 반영 범위: ~M/D. 다음 정리 시 이 날짜 이후 메시지만 추가 반영할 것)` 형식의 인용구 한 줄 + 바로 아래 `<table_of_contents/>` 블록(2-9 참고)을 넣고, 그 뒤로 `### 전체 흐름` → `### Q&A (...)` → `### 참고 이미지`(있는 경우) → `### 공유된 링크`(있는 경우) → `### 토론 요약` → `### 새로운 용어 설명` → `### 주요 참여자 유형` → `### 실무 팁 / 활용법`(있는 경우) 순서로 전체 섹션을 다 넣는다.
   - **★ 도구 호출 함정**: `notion-create-pages`/`notion-update-page`는 `parent`, `pages`, `content_updates` 등을 **네이티브 JSON 파라미터**로 직접 넘겨야 한다. 통짜 JSON 문자열을 하나의 `raw`/`input` 필드에 욱여넣는 식으로 호출하면 `InputValidationError`가 난다(이번 세션에서 실제로 겪음) — 각 파라미터를 개별 필드로 정확히 채워서 호출할 것.
2. **인덱스 페이지에 링크 삽입**: `notion-create-pages`로 새 페이지를 만들면 그 `<page url="...">` 블록이 **인덱스 페이지 맨 끝에 자동으로 붙는다** (자식 페이지이므로). 이걸 원하는 번호 위치로 옮기려면, 인덱스 페이지를 `replace_content`(또는 `update_content`)로 다시 쓰면서 **이미 있는 `<page url="...">` 태그를 그대로 원하는 줄로 옮겨 붙인다** — 새로 만든 페이지를 일반 마크다운 링크(`[제목](url)`)로 쓰면 "이 작업은 자식 페이지를 삭제하게 된다"는 `validation_error`가 난다(자식 페이지 블록은 `<page url>` 태그로만 유지 가능, 손자병법 파이프라인에서도 동일하게 확인된 규칙). 인덱스 페이지가 짧으므로(방 10줄뿐) `replace_content`로 통째로 다시 쓰는 게 가장 안전하다.
3. **재확인**: 새 하위 페이지와 인덱스 페이지 둘 다 `notion-fetch`로 재조회해서 한글 깨짐·마크다운 깨짐(특히 URL을 괄호 안에 맨텍스트로 쓴 경우)이 없는지 확인한다.

과거 방식(인덱스 페이지 안에 `## N. 방이름` 헤더로 전부 몰아쓰던 방식)은 폐기됐다 — 방이 늘어날수록 인덱스 페이지 자체가 비대해지는 문제로 하위 페이지 분리로 전환함.

## 4. 재사용 스크립트

- `카톡정리/parse_kakao.py` — 파서 + Q&A 후보 추출기. 새 zip이 추가되면 그대로 재사용.
- 사람(세션의 Claude)이 직접 해야 하는 부분: 운영진 이름 판별, 전체 흐름 서사 작성, Q&A 후보 중 진짜 질문 선별 + 중복 병합 + 답변 통합, 노션 토글 작성. **이 부분은 규칙 기반으로 자동화할 수 없다** — 의미 판단이 필요하기 때문에 매번 세션이 직접 읽고 판단해야 한다.

## 5. 주간 증분 업데이트 (2회차부터 적용)

**★ (사용자 명시 요청, 2026-07-23 확정) 이 정리는 일주일에 한 번씩 반복한다.** 같은 방의 새 zip이 다시 올라오면(카카오톡 대화 내보내기는 항상 처음부터 전체 대화가 담겨 있음), 전체를 다시 쓰지 않고 **이전 정리 이후의 새 내용만 이어붙인다.**

1. **기준 날짜 확인**: 그 방의 노션 페이지를 `notion-fetch`로 열어서 맨 위 `> 📅 마지막 정리 기준일: YYYY-MM-DD` 를 확인한다. 이게 이번 회차에서 "여기부터 새 내용"의 기준이 된다.
2. **새 zip 추출·파싱**: 1\~2번 절차와 동일하게 새 zip을 압축 해제하고 `parse_kakao.py`로 파싱한다(전체 대화가 다시 담겨 있으므로 결과 `messages.json`도 전체 기간을 포함함).
3. **신규 구간만 필터링**: `messages.json`에서 날짜가 마지막 정리 기준일보다 **이후인 메시지만** 추려서 이번 회차 분석 대상으로 삼는다(날짜 형식 `M/D/YY` 파싱 필요). qa_candidates도 이 신규 구간에 한정해서 다시 뽑는다.
4. **섹션별로 "이어붙이기"**(2번 섹션의 각 하위 절차를 신규 구간에만 적용) — **절대 기존 내용을 통째로 지우고 다시 쓰지 않는다**:
   - **전체 흐름**: 새 시기에 해당하는 불릿 항목(들)을 기존 불릿 리스트 끝에 추가하고, 맨 아래 "한 줄 요약"만 필요시 갱신한다. `notion-update-page`의 `update_content`로 기존 "한 줄 요약" 줄을 찾아서 그 앞에 새 불릿을 삽입.
   - **Q&A**: 신규 구간에서 나온 질문이 기존 토글과 같은 주제면 **그 토글의 답변 내용을 갱신**(새로 알게 된 정보 반영), 완전히 새로운 주제면 **새 토글을 Q&A 섹션 끝에 추가**한다.
   - **공유된 링크 / 토론 요약 / 새로운 용어 설명 / 실무 팁**: 신규 구간에서 나온 것만 추려서 기존 리스트/토글 뒤에 추가(중복 URL·중복 용어는 스킵).
   - **주요 참여자 유형**: 기존 유형 분류가 여전히 맞으면 그대로 두고, 신규 구간에서 새롭게 두드러진 헤비유저/유형이 있을 때만 추가.
   - **참고 이미지**: 신규 zip에 새 이미지가 있으면 2-3 절차(PII 스크리닝 → 리사이즈 → 커밋 → GitHub raw URL 임베드)를 그대로 반복해서 기존 이미지 목록 뒤에 추가.
5. **기준 날짜 갱신**: 모든 섹션 업데이트가 끝나면 페이지 맨 위 `마지막 정리 기준일`을 이번 회차에서 반영한 가장 최신 메시지 날짜로 `update_content`로 교체한다.
6. **재확인**: `notion-fetch`로 다시 열어서 중복 삽입·깨짐이 없는지 확인하고, README의 5번(완료 현황) 표에서 해당 방의 "마지막 정리일"도 갱신한다.

`카톡정리/`와 `북마크관리/`는 이 저장소 안에서 서로 다른 독립 프로젝트다. 북마크관리는 크롬 북마크 점검 도구(`fix_bookmarks.py`)이고, 카톡정리는 이 문서가 다루는 카카오톡 대화 분석이다. 혼동하지 말 것.
