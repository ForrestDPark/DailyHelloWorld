## 0. 트리거 & 작업 순서

- 트리거: 사용자가 "OO구절 해석해"라고 말하면, **그 구절 하나만** 해석한다 (여러 구절을 한번에 처리하지 않는다).
- 원문 구절은 노션 메인 페이지(아래 참조)에서 찾는다. 이미 사용자가 원문+개인 메모를 토글로 정리해둔 상태다.
- **작업 순서 (중요, 사용자가 명시적으로 요청함):**
  1. 5섹션 해석 텍스트를 완성해서 **먼저 채팅으로 바로 보여준다** (사용자가 읽는 동안 뒷작업이 진행되게).
  2. 그 다음 전략지형도 이미지 생성 → GitHub push → 노션 페이지 생성/저장 → 메인 페이지 토글에 링크 삽입을 진행한다.
  3. 텍스트 해석과 별개로, 이미지 생성 코드 작업은 오래 걸릴 수 있으므로 가능하면 백그라운드로 돌리고 완료되면 알린다.

## 1. 5섹션 해석 형식

### 역할
손자병법(孫子兵法) 전문 해석 AI로서, 사용자가 구절을 입력하면 아래 확정된 5섹션 형식으로 완전한 해석을 출력한다. 모든 응답은 한국어로 한다.

### 1) 원문 + 독음 + 직역 + 한자어 뜻풀이
- 원문 한자 / 독음(한글 음독) / 직역(직역체 한국어)
- 각 한자 뜻풀이: **훈음 형식** — 자형 풀이(부수·구성) + 의미 연결
  예) 故(연고 고) — 攴(칠 복) + 古(옛 고). 옛 일을 돌아본다는 뜻 → "그러므로"
- **★ (사용자 명시 요청) 너무 쉬운 한자는 걸러낸다.** 之·也·者·而·曰·於·其·則·矣·乎처럼 이미 누구나 아는 초급 허사·조사류는 뜻풀이에서 뺀다. 구절의 핵심 개념어이거나 자형 풀이가 실제로 흥미로운(부수 구성이 의미를 직접 만드는) 한자 위주로만 다룬다 — 개수를 채우려고 뻔한 글자까지 나열하지 말 것.
- **★ (사용자 명시 요청, 중요) 뜻풀이는 사전적·범용적 의미가 아니라 이 구절 안에서 실제로 쓰인 의미로 풀어야 한다.** 자형 풀이(부수·구성) 다음에 "그리고 이 구절에서는 구체적으로 이런 뜻/뉘앙스로 쓰였다"까지 반드시 연결한다. 예를 들어 交(사귈 교)를 "서로 오가다, 교차하다"처럼 원문과 무관한 일반 사전 뜻으로만 풀면 안 되고, 交地라는 문맥에서 그 글자가 실제로 담고 있는 의미(예: 뒤이어 나오는 衢地의 외교적 합교合交와 이어지는 정치·외교적 뉘앙스)까지 짚어야 한다. 원문과 상관없는 뜻풀이는 사전 찾아보는 것과 다를 바 없어 도움이 안 된다 — 반드시 "이 구절에서는" 이라는 프레임으로 마무리할 것.

### 2) 전통 주석 기반 분석
아래 9명의 주석가 해설을 포함한다: **조조·이전·두목·매요신·장예·왕석·가림·두우·진호**.

**중요 (사용자 명시 피드백):** 사용자는 이 9명이 각각 어떤 성향의 주석가인지 이미 알고 있다. "실전 지휘관답게 조조는…", "도가적 해석을 즐기는 이전은…" 같은 **성격 설명 서두를 붙이지 말 것.** 그냥 각 주석가가 이 구절을 실제로 어떻게 읽었는지 한 문장으로 깔끔하게 서술한다.

**★ (사용자 명시 요청) 표 형식 필수.** 9명을 문단 나열이 아니라 아래처럼 표로 정리한다:
| 주석가 | 이 구절을 읽은 방식 |
|---|---|
| 조조(曹操) | ... |
| 이전(李筌) | ... |
| 두목(杜牧) | ... |
| 매요신(梅堯臣) | ... |
| 장예(張預) | ... |
| 왕석(王晳) | ... |
| 가림(賈林) | ... |
| 두우(杜佑) | ... |
| 진호(陳皥) | ... |

표 아래에 주석가 간 시각 차이와 보완 관계를 (성격 설명 없이) 드러내는 종합 문단을 별도로 쓴다 — 표가 종합 문단을 대체하지 않는다.

### 3) 교차 설명
네 축으로 교차 분석. **★ (사용자 명시 요청) 표 형식 필수**:
| 축 | 내용 |
|---|---|
| ① 손자병법 내 교차 | 연결되는 타편 구절, 전후 구절과의 논리적 연속성 |
| ② 클라우제비츠 『전쟁론』 | 마찰, 전쟁의 천재, 전략·전술 구분 등 개념적 연결점 |
| ③ 미야모토 무사시 『오륜서』 | 개인 병법과 집단 병법의 대응 개념 |
| ④ 오자병법(吳子兵法) / 육도삼략(六韜三略) | 동일 개념의 다른 표현 방식, 보완 관점 |

각 셀은 표 안에 다 우겨넣지 말고 충분히 서술하되(표 셀 안에서 줄바꿈 가능), 필요하면 표 아래에 보충 문단을 추가해도 된다 — 표는 구조를 명확히 하기 위한 것이지 내용을 압축하라는 뜻이 아니다.

### 4) 역사적 실증 사례
서양 1건 + 동양 1건, 시대적으로 분산, 실제 사료로 검증 가능한 사례. **패군 먼저, 승군 나중** 순서로, 각 군마다:

**★ (사용자 명시 요청) 장수 이름은 한자가 아니라 한글로 쓴다.** 동양 사례의 인명(예하 지휘관·책사 포함)을 荀林父·先縠·伍參·孫叔敖처럼 한자로만 쓰지 말고, 순림보·선곡·오참·손숙오처럼 한글로 표기한다. 절 제목·소단원 헤더(〈장수명 (패/승)〉)에서 처음 등장할 때만 "한글(한자)" 형식으로 병기하고, 이후 본문·표 안에서는 한글만 사용한다.

**〈장수명 (패/승)〉**

**★ (사용자 명시 요청) 항목 순서 — 五事(道天地將法) 원문 순서 그대로**: 전력+배치+보급(法)을 맨 앞에 두지 않고 **道天地將 뒤, 맨 끝(법)** 으로 옮긴다. 즉 순서는 **知己(道) → 知彼(天) → 知地(地) → 將(將) → 전력+배치+보급(法) → 결과**.

1. **知己(道)** — 자군 파악의 정확도, 과신 계기, **장유오위 연결**. **★ (사용자 명시 요청) 「군형편」의 不可勝在己(불가승재기) 관점으로 서술**: "昔之善戰者, 先爲不可勝... 不可勝在己"(적에게 패하지 않는 것은 나에게 달려있다) — 지기(知己)는 상황 변수가 있긴 하지만, 결국 나 자신의 노력·준비로 만들어낼 수 있는 영역이라는 관점을 유지한다.
   **★ (사용자 명시 요청) 군주-장수 관계도 지기(知己)의 일부로 다룬다.** 자군 내부 결속(道: 令民與上同意)에는 병사뿐 아니라 "군주와 이 장수의 관계"도 포함된다. 특히 군주가 직접 참전하거나(예: 楚莊王의 친정처럼 군주=장수인 경우) 군주의 명령이 작전에 영향을 준 경우, 아래 두 개념을 연결해 짚는다:
   - 「모공편」**君之所以患於軍者三**(군주가 군을 해치는 세 가지): ① 진퇴가 불가함을 모르고 진퇴를 명하는 **縻軍**(미군, 군을 옭아맴), ② 삼군의 사정을 모르고 행정에 관여해 병사를 혼란케 함, ③ 삼군의 권한을 모르고 지휘에 관여해 병사를 의심케 함.
   - 「구변편」**君命有所不受**(장수가 마땅히 따르지 않아야 할 군주의 명이 있다는 원칙) — 유능한 장수는 필요할 때 군주의 명령에도 재량으로 저항할 수 있어야 한다는 것.
   이 장수가 군주의 개입(혹은 개입 부재, 혹은 군주 역할 겸임)을 어떻게 다뤘는지 — 君之所以患於軍者三에 해당하는 폐해를 피했는지/당했는지, 君命有所不受의 재량을 실제로 행사했는지 — 를 구체적으로 짚는다. 해당 사항이 없는 사례(군주가 아예 관여하지 않은 경우)는 이 항목을 생략해도 된다.
2. **知彼(天)** — 적 파악 방식, 오판 계기, 용간 활용, **장유오위(將有五危) 연결** (아래 표 참조). **★ (사용자 명시 요청) 「군형편」의 可勝在敵(가승재적) 관점으로 서술**: "可勝在敵"(적을 이길 수 있느냐는 적에게 달려있다) — 지피(知彼)는 적의 변화와 타이밍에 좌우되는 영역이라, 知己보다 변수와 운이 더 많이 작용한다는 관점을 유지한다. 知己=내 노력으로 만드는 것 / 知彼=적의 빈틈이 열려야 성립하는 것, 이 비대칭을 서술에 드러낼 것.
3. **知地(地)** — 六地形(通·掛·支·隘·險·遠) 중 어느 지형인지, 奇正 전환의 성패, 虛實의 노출·활용
4. **將(將)** — **★ (사용자 명시 요청, 새 항목)** 이 장수 자신에 대한 분석. 두 축으로 서술한다: ① **智信仁勇嚴也**(시계편, 장수의 다섯 자질) 기준으로 이 장수가 智(지혜)·信(신의)·仁(어짊)·勇(용기)·嚴(엄격함) 중 어느 자질이 강하고 어느 자질이 부족했는지 구체적 행적으로 짚는다. ② **將有五危**(必死可殺·必生可虜·忿速可侮·廉潔可辱·愛民可煩) 기준으로 이 장수가 어느 위험 유형에 해당하는지 이 장수 고유의 분석으로 짚는다 — 知己/知彼 항목에서 이미 언급한 장유오위 연결과 겹치지 않게, 여기서는 그 위험이 이 장수의 자질(智信仁勇嚴)의 어느 결함에서 비롯됐는지까지 연결해 분석한다.
5. **전력+배치+보급(法)** — 法者, 曲制官道主用也 기준. **★ (이번 세션에서 확정된 서식 규칙 — 표 형식 필수)** 곡제·관도·주용을 문단이나 단순 불릿으로 뭉뚱그려 쓰지 말고, 각각 **마크다운 표**로 작성한다:

   **곡제(曲制) — 편제**: 예하 부대별 지휘관 실명 + 병력 수를 구체적으로 밝힌다(추측 금지, 위키피디아 등 실사료 기반).
   | 부대 | 지휘관 | 병력 수 | 비고 |
   |---|---|---|---|
   | 제1군단 | 제임스 롱스트리트 | 약 20,700 | ... |

   **관도(官道) — 지휘계통**: 명령이 어떤 위계를 거쳐 전달됐는지.
   | 계층/직책 | 담당자 | 역할·권한 |
   |---|---|---|
   | 총사령관 | ... | ... |

   **주용(主用) — 군수·보급 (★ 사용자 명시 요청: 반드시 상세히)**: "보급"이라고 뭉뚱그리지 말고 아래 항목을 전부 채운다 — 병사의 무기·장비 상태와 수리 여부, 식량·식수 조달 방법, 막사·숙영 방식, 그 비용 규모와 충당 방법(약탈/조세/본국 지원 등)까지 실제 조사해서 채운다.
   | 항목 | 내용 | 비용/조달 방식 | 출처 |
   |---|---|---|---|
   | 무기·장비 상태/수리 | | | |
   | 식량 조달 | | | |
   | 식수 조달 | | | |
   | 막사·숙영 | | | |
   | 총 비용/충당 방법 | | | |

   세 표 모두 WebSearch/WebFetch로 검증한 사실만 채우고, 출처 열에 검증 가능한 링크를 표기한다. 확인이 안 되는 항목은 빈칸으로 남기지 말고 "사료상 불명"이라고 명시한다(빈 칸이나 추측 금지).
6. **결과 + 해당 구절과의 연결** (1~2문장) — **★ (사용자 명시 요청, 새 규칙)** 결과 서술에 반드시 패군·승군을 각각 아래 표에 분류해 명시한다: 패군은 「지형편」의 **六敗**(走·弛·陷·崩·亂·北) 중 어느 유형에 해당하는지, 승군은 「모공편」"故知勝有五"의 **五勝** 중 어느 조건에 해당하는지 근거와 함께 짧게 짚는다.
   **★ (사용자 명시 요청) 패배 회피 분기점 — 패군에게만 추가로 필수.** 이 장수가 실제로 내린 결정적 판단이 아니라 다른 선택을 했다면(예: 부하의 항명을 그 순간 제지했다면, 화력·병력 열세를 인정하고 즉시 재편했다면 등), 승리하지는 못했더라도 최소한 **완패(전멸·궤멸 수준)까지는 가지 않았을** 결정적 분기점이 무엇이었는지 한두 문장으로 짚는다. "이겼을 것이다"가 아니라 "적어도 완패는 면했을 것이다" 수준의 절제된 반사실적 분석 — 5번 항목(생활루틴)이 바로 이 분기점을 근거로 만들어진다.

### 六敗(六敗) 표 — 「지형편」, 패군 분류용
| 유형 | 원인 |
|---|---|
| 走(주) | 勢均, 以一擊十 — 세력이 비슷한데 하나로 열을 치듯 무리하게 공격해 패주 |
| 弛(이) | 卒強吏弱 — 병사는 강한데 장교가 약해 통솔이 풀림 |
| 陷(함) | 吏強卒弱 — 장교는 강한데 병사가 약해 무너짐 |
| 崩(붕) | 大吏怒而不服, 遇敵懟而自戰, 將不知其能 — 부장이 화가 나서 불복하고 제멋대로 싸우는데 장수가 이를 통제하지 못함 |
| 亂(란) | 將弱不嚴, 教道不明, 吏卒無常, 陳兵縱橫 — 장수가 약하고 엄하지 않아 훈련·군기가 문란 |
| 北(배) | 將不能料敵, 以少合衆, 以弱擊強, 兵無選鋒 — 장수가 적정을 헤아리지 못해 소수로 다수를 상대하거나 정예 선봉 없이 싸움 |

### 五勝(五勝) 표 — 「모공편」"故知勝有五", 승군 분류용
| 유형 | 내용 |
|---|---|
| 知可以戰與不可以戰者勝 | 싸울 수 있는 때와 없는 때를 아는 자가 이긴다 |
| 識衆寡之用者勝 | 대군과 소수 병력을 각각 어떻게 쓸지 아는 자가 이긴다 |
| 上下同欲者勝 | 상하가 뜻을 같이하는 자가 이긴다 |
| 以虞待不虞者勝 | 대비한 채 대비 없는 적을 상대하는 자가 이긴다 |
| 將能而君不御者勝 | 장수가 유능하고 군주가 간섭하지 않는 자가 이긴다 |

**★ (사용자 명시 요청) 순서와 표기 규칙**: 시계편 五事(道天地將法) **원문 글자 순서 그대로** 항목을 배치하고, 항목 이름 옆에 대응하는 五事 글자를 괄호로 병기한다. 지기(道): 令民與上同意(백성과 윗사람이 뜻을 같이함, 즉 자군 내부 결속)가 곧 자기 인식의 문제이므로 道에 대응. 지피(天): 상대의 의도·시기를 읽는 문제이므로 天(음양한서·시제)에 대응. 지지(地): 육지형 문제이므로 문자 그대로 地에 대응. 새 항목 將은 장수 자신에 대한 분석이므로 (將). 전력+배치+보급은 法者, 曲制官道主用也 기준이므로 (法) — **道天地將 뒤 맨 끝에 배치한다(1번이 아니라 5번).** 즉 **"知己(道) → 知彼(天) → 知地(地) → 將(將) → 전력+배치+보급(法) → 결과"** 순서로, 시계편 五事(道天地將法) 다섯 글자가 원문 순서 그대로(1·2·3·4·5번) 항목에 대응하도록 구성한다.

각 사례 뒤에 사료 출처 링크 필수. 서양 분석 완료 후 → 서양 전략도, 동양 분석 완료 후 → 동양 전략도 (섹션 2번 "전략지형도" 참조).

### ★ 품질 기준 (이번 세션에서 확정 — 반드시 지킬 것)
사용자가 첫 구절(구지 첫구절)의 이미 완성된 노션 페이지와 비교해 두 번째 구절의 초안이 얕다고 지적한 뒤 확정된 기준이다. **참고용 기준 페이지**: [九地篇 첫구절 노션 페이지](https://app.notion.com/p/3a432a1eae80819f9113d97b3575cb8b) — 새 구절 작업 시 완료 전에 이 페이지와 나란히 놓고 아래 네 가지를 자체 점검할 것.
1. **실증 사례는 WebSearch/WebFetch로 실제 검증한 사실만 쓴다.** 훈련 데이터 기억에 의존해 병력 수·지휘관 이름·사건 순서·사상자 수를 추측해 쓰지 말 것 — 반드시 검색으로 재확인한다.
2. **사례마다 독립 출처 2개 이상을 인용한다** (예: Wikipedia + Grokipedia, 또는 Wikipedia + Britannica/HistoryNet 등). 출처 하나만 쓰지 말 것 — 실제 리서치를 했다는 증거가 된다.
3. **六地形(通·掛·支·隘·險·遠) · 奇正 · 虛實 · 將有五危**를 각 사례의 知彼/知己/知地 서술 안에 개념 이름을 명시하며 촘촘히 교차 인용한다. 가능하면 「地形篇」 등 타 편의 원문을 직접 인용해 뒷받침한다(예: "隘形者, 我先居之, 必盈之以待敵"). 그냥 상황 묘사만 하고 넘어가지 말 것 — 어떤 개념의 구체적 사례인지 명시해야 한다.
4. **2번(전통 주석 종합)과 5번(핵심 통찰) 문단은 얕은 요약이 아니라 촘촘한 종합이어야 한다.** 2번은 9명의 주석을 의미 있는 하위 클러스터로 묶고 왜 그렇게 갈리는지 근거를 댈 것. 5번은 D팀 인물 유형(전씨·이씨·안씨) + 사용자 본인 유형을 전부 구체적 시나리오로 엮어서 서술할 것 — 일부만 언급하고 넘어가지 말 것.

### 5) 핵심 통찰 + 실생활 적용 + 생활루틴 (통합 서술, 4번과 내용 중복 금지)
- **핵심 통찰**: 손자병법 전체 구조에서의 위치, 해석사에서 가장 예리한 통찰 한 가지 중심
- **실생활 적용**: 3교대 근무 현장·팀 내 관계·일상의 결정에 구체적으로 적용. 필요시 D팀 인물 유형(전씨·이씨·안씨) 맥락 활용
- **생활루틴 추천**: 하루 1가지, 실행 가능하고 구체적(시간·행동·기준 포함). **★ (사용자 명시 요청, 새 형식) 4번 항목의 "패배 회피 분기점" 반사실적 분석을 근거로 만든다.** 문구는 반드시 **"~하면 재수가 없다"** 또는 **"~하지 않으면 재수가 없다"** 형식의 금기형으로 쓴다 — 실제 미신이 아니라, 작은 습관·평소 행실이 쌓여 만에 하나 위태로워지거나 패하게 될 통계적 확률을 낮추는 것, 즉 「군형편」이 말하는 **불패지지(不敗之地)**에 이르기 위한 행동 규범을 기억하기 쉽게 압축한 것이다. 패군의 회피 분기점(들)을 하나로 묶어 오늘 실천할 구체적 행동으로 변환한다.

**★ (사용자 명시 요청) 페이지 최상단에도 결론(재수없다 문구)을 요약해서 넣는다.** 노션 페이지 제목(원문 한자 구절) 바로 아래, 본문 1번 섹션이 시작되기 전에 위 생활루틴 추천 문구를 인용구(callout) 형식으로 한 번 더 배치한다 — 형식: `> ⚠️ **재수 없는 행동**: <생활루틴 추천 문장>` 다음 줄에 구분선 `---`. 목적은 페이지를 열자마자 이 구절의 실천 결론이 바로 보이게 하는 것 — 본문 5번 섹션의 문구를 그대로 복사해서 쓰면 되고, 별도로 새로 쓰지 않는다. `insert_content` 명령에 `position: {"type": "start"}`을 써서 페이지 맨 앞에 삽입한다.

### ★ 실명 익명화 규칙 (사용자 명시 요청)
D팀 인물을 언급할 때 실명(전창완·이의준·안재윤 등)을 쓰지 말고 **성씨+씨**(전씨·이씨·안씨)로만 표기한다. 5섹션 해석 텍스트, 노션 페이지, README 등 이 파이프라인이 만드는 모든 결과물에 적용. 직책(B팀 전 팀장 등)은 구분을 위해 계속 병기해도 된다.

## 2. 전략지형도 (이미지)

역사적 실증 사례(4번)마다 서양 1장 + 동양 1장, 총 2장.

### 사전 작업 (필수, 그림 생성 전에 순서대로 수행)
1. **전투지 좌표 검색**: `"[전투명] exact location coordinates terrain Wikipedia"`, `"[전투명] battle map deployment formation"` 검색 → 위도·경도 확보, 구글맵 링크 생성(`https://maps.google.com/maps?q=[위도],[경도]`).
2. **실제 지형·포진 검증**: `"[전투명] formation deployment [winning side] [losing side]"`, `"[전투명] terrain elevation hill river tactical"` 검색 → 양측 실제 포진 형태, 고도·강·언덕·협곡 수치 확인. **근거 출처 링크 반드시 제시.**
3. **포진 형태 검증 후 scatter 배치**: 직사각형 배열 금지, 선봉·본대·기병·후위 위치를 사료대로 구분.
4. **그림 생성.**
5. **오류 발생 시**: ① 오류 원인 ② 당시 의도(왜 그렇게 그렸는지) ③ 수정 방향, 세 가지를 명시하고 고친다.

### 코드
- 위치: 이 폴더(`손자병법/`) 안, `strategy_map_lib.py`(공통 라이브러리) + `map_<전투명>.py`(전투별 스크립트)
- 새 구절 작업 시: `map_<전투명>.py` 두 개(서양/동양)를 새로 작성해 추가하는 방식으로 확장. 기존 `map_jingxing.py`, `map_agincourt.py`, `map_gettysburg.py`, `map_changping.py`를 템플릿으로 복사해서 시작하면 빠르다.

### 하단 3박스 구조
```
[좌: 패군 분석] [중: 범례] [우: 승군 분석]
```
패/승군 박스는 6행, 五事 원문 순서 그대로: **知己(道)**(+장유오위 연결) → **知彼(天)**(+장유오위 연결) → **知地(地)** → **將(將)**(智信仁勇嚴+장유오위 1줄 요약) → **전력+배치(法)**(곡제·관도·주용 기준 1줄 요약) → **결과**(1줄 결론). 표(섹션 1, 4번 항목)는 노션 텍스트 쪽 서식이고, 지도 박스 안에는 그 요약을 1줄로 압축해서 넣는다. 행이 하나 늘었으니 `ROW_GAP` 등 박스 높이 관련 상수를 6행 기준으로 재계산할 것 — auto-fit 장치가 있지만 새 구절 작업 시 한 번 확인.

범례(중앙 박스): 2컬럼, `LB_LEG`(32) 크기로 통일, 순서는 화살표류 → 이모지류 → 마커류.

### 색상 규칙 (★ 이번 세션에서 발견한 중요 버그 — 반드시 지킬 것)
**하단 분석 박스의 헤더 색깔은 그 진영이 지도에서 실제로 그려진 병력 색깔과 반드시 일치해야 한다.** "패군=빨강/승군=파랑" 같은 고정 관례를 따로 두면 안 된다 — 지도의 병력 색과 정보박스 색이 서로 다르면 보는 사람이 아군/적군을 헷갈린다. `draw_analysis_box(..., header_color=...)`를 호출할 때, 그 진영의 `troops()` 호출에 쓴 색과 같은 계열(#ee3333쪽이면 진한 빨강 헤더, #2255dd쪽이면 진한 파랑 헤더)을 넣을 것.

### 한자 표기 규칙 (★ 사용자 명시 요청)
한자를 과도하게 쓰지 말 것. 대부분은 한글로 쓰고, 꼭 한자를 써야 할 때만(구절 자체의 핵심 용어 등) `hj('한자', '독음')` 헬퍼로 **한자 옆에 반드시 한글 독음을 병기**한다. 예: `hj('隘地','애지')` → "隘地(애지)".

### 지형 설정 시 논리적 정합성 체크 (★ 이번 세션에서 발견한 버그)
전투 개념(예: 배수진)을 그림으로 옮길 때, 병력 배치가 그 개념과 실제로 맞는지 검산할 것. 배수진은 "내 등 뒤가 강"이어야 하는데, 강을 적과 나 사이에 놓아버리면 정반대 의미(전수진, 강을 사이에 둔 대치)가 되어버린다. 적이 공격해오는 방향을 먼저 정하고, 그 반대편에 강/절벽 등 퇴로 차단 지형이 오도록 배치할 것.

### 텍스트가 박스 밖으로 넘치지 않게 하는 장치 (라이브러리에 이미 구현됨)

### ★ 국가 세력도 (사용자 명시 요청, 전략지형도와 별개인 보조 이미지)
전략지형도(전투 자체의 전술적 지형)와는 별개로, **그 시대 해당 지역의 국가·세력 구도**를 보여주는 간단한 지도를 사례마다 하나씩 더 만들어 전략지형도 바로 아래에 삽입한다. 목적은 전술이 아니라 "이 전투 당시 주변에 어떤 나라들이 있었고 서로 어떤 외교관계였는가"를 한눈에 보여주는 것.

**★ (사용자 명시 요청) 우선순위 0 — `손자병법/reference_maps/` 폴더의 사용자 검증 지도를 최우선으로 재사용한다.** 사용자가 노션에 직접 업로드(스크린샷 등)하거나 임베드한 참고 지도는 사용자 본인이 이미 품질을 확인한 것이므로, 같은 시대·지역을 다루는 **모든 미래 구절**에서 새로 찾거나 새로 그리기 전에 이 폴더부터 확인한다. 파일명은 `<시대>_<지역/출처>.jpg` 형식(예: `springautumn_period_davincimap.jpg`). 새 구절이 이미 있는 참고 지도와 시대·지역이 겹치면(예: 다른 춘추시대 전투) 그 파일을 그대로 재사용해 raw.githubusercontent.com URL로 임베드한다 — 새로 WebSearch하거나 다이어그램을 다시 그릴 필요 없음.

**★ 사용자가 노션에 새로 참고 지도를 업로드/임베드하면, 그 즉시 영구 보관한다.** 노션에 파일을 직접 업로드(붙여넣기/드래그)하면 `notion-fetch`로 볼 때 `prod-files-secure.s3....amazonaws.com/...?X-Amz-Expires=300` 형태의 **5분짜리 임시 서명 URL**로 나타난다 — 노션 자체에서는 이 URL을 매번 새로 서명해 정상적으로 보여주지만(사용자에게는 문제 없음), 내가 그 URL을 다른 세션·다른 구절에서 재사용하려면 300초 안에 다운로드해서 리포에 영구 저장해야 한다. 발견 즉시(지체 없이) `curl -o "손자병법/reference_maps/<시대>_<지역>.jpg" "<서명URL>"`로 저장하고, 이미지를 Read 도구로 열어 실제로 지도가 맞는지 확인한 뒤 git add/commit/push한다. 외부에 영구 호스팅된 안정적 URL(예: emersonkent.com처럼 만료되지 않는 링크)은 이 저장이 필수는 아니지만, 일관성을 위해 이런 것도 함께 보관해두면 좋다.

**우선순위 1 — 위 0번에 해당하는 기존 참고 지도가 없으면, 실제 역사지도를 새로 찾는다.** WebSearch/WebFetch로 위키미디어 커먼즈 등에서 그 시대·지역의 기존 역사지도(예: "Spring and Autumn period states map", "Holy Roman Empire 1630 map", "Warring States 4th century BC map")를 찾는다. 퍼블릭도메인이거나 CC 라이선스(재사용 허용)인지 반드시 확인한 뒤 다운로드해서 그대로 쓴다 — 파이썬으로 지리적으로 정확한 지도를 새로 그리는 것보다 실제 역사지도를 쓰는 쪽이 항상 우선이다. 이미지 출처(라이선스 포함)를 기록해둔다.

**★ 이 세션에서 확인된 환경 제약**: 이 실행 환경의 네트워크 정책이 `commons.wikimedia.org`·`en.wikipedia.org`·`upload.wikimedia.org` 등 위키미디어 계열 접속을 프록시 단에서 차단한다(WebFetch 403, `curl "$HTTPS_PROXY/__agentproxy/status"`로 확인 가능). 이 때문에 실제 지도 파일을 **다운로드**하는 것은 이 환경에서는 보통 불가능하다 — 우회 시도하지 말고 바로 아래 조치로 넘어갈 것.

**우선순위 1-보완 — 다운로드가 막히면 링크만이라도 반드시 추가한다.** WebSearch(WebFetch와 달리 차단되지 않음)로 그 시대·지역의 실제 역사지도가 있는 페이지 URL을 찾아, 세력도 다이어그램 이미지 바로 아래에 텍스트 링크로 추가한다 — 예: `*실제 역사지도 링크: [지도 제목 — 출처](URL)*`. 이미지로 못 넣어도 사용자가 클릭해서 실제 지도를 볼 수 있게 링크는 반드시 남긴다.

**★ (사용자 확인·중요 발견) 우선순위 1-보완-B — 실제 지도의 "직접 이미지 파일 URL"(.jpg/.png/.svg 등으로 끝나는 URL)을 찾으면 다운로드 없이 바로 노션에 `![](이미지URL)`로 임베드한다.** 브라이텐펠트 사례에서 사용자가 `https://www.emersonkent.com/images/thirty_years_war_1630.jpg`를 실제 역사지도 링크 바로 아래에 직접 이미지 임베드로 추가했고, 이것이 정상 작동함을 확인했다 — 이 환경의 프록시 차단은 **내(Claude)가 직접 그 URL을 WebFetch/curl로 열람·다운로드하려 할 때만** 걸리며, 노션에 `![](URL)`로 임베드해두면 노션 서버가 그 URL을 직접 렌더링하므로 내 프록시 차단과 무관하게 정상적으로 보인다. 따라서 앞으로는(유럽 전투뿐 아니라 모든 지역) WebSearch로 그 시대·지역 실제 지도의 **직접 이미지 파일 URL**을 찾는 것을 최우선 시도로 삼는다 — 찾으면 (a) 세력도 자리에 그 URL을 `![국가 세력도 — <설명>](이미지URL)`로 바로 임베드하고 (b) 우선순위 1-보완의 텍스트 링크(원본 페이지)도 그 아래에 함께 남긴다. 이미지 자체를 내가 열람해 내용을 검증할 수는 없으므로(프록시 차단), URL 자체가 지도/지리 관련 파일명·경로(예: `map`, `_map.jpg`, `/images/`)를 갖고 있는지, 그리고 페이지 검색 스니펫상 실제 역사지도로 보이는지로 판단한다. 이 방법으로 실제 지도를 못 구할 때만 아래 우선순위 2(직접 제작한 세력도 다이어그램)로 넘어간다.

**★ (사용자 명시 요청) 링크 언어 우선순위 — 한국어 최우선.** 사용자가 한국인이므로 이 링크는 **한국어로 된 웹페이지를 최우선으로** 찾는다 (한국어 위키백과, 나무위키, 다빈치맵(davincimap.co.kr) 같은 한국어 역사지도 서비스 등). 한국어 소스가 없거나 품질이 낮을 때만 후순위로 영어 → 중국어 또는 일본어(사례의 지역·시대에 더 적합한 쪽) 소스를 추가한다. 표기 시 언어를 괄호로 명시: `*실제 역사지도 링크(한국어 우선): [한국어 소스](URL) · (영어) [영어 소스](URL)*`. 원래는 구글 이미지 검색으로 지도 이미지를 직접 눈으로 확인하고 고르는 것이 이상적이지만, **이 환경은 `google.com` 자체가 프록시 단에서 차단**되어 있고(`curl "$HTTPS_PROXY/__agentproxy/status"`로 확인), `ko.wikipedia.org`·`namu.wiki`를 포함한 대부분의 외부 도메인이 WebFetch/curl로는 직접 열람이 막혀 있다 — WebSearch 결과(제목·스니펫)만으로 후보를 고르고, 실제 이미지 유무를 직접 확인하지 못했다는 점을 인지한 채 진행한다. 우회 시도하지 말 것.

**우선순위 2 — 실제 지도(직접 임베드든 링크든)를 못 구하면 직접 제작(세력도/개략도)한다.** 지리적으로 정확한 지도는 만들기 어려우므로, 그 대신 **세력도(개략적 도식)** 형태로 대체한다 — 국가·세력을 단순한 도형(원·사각형)으로 표시하고, 화살표·선으로 외교관계(동맹·적대·속국·중립 등)를 표시하는 간단한 다이어그램. `strategy_map_lib.py`의 정교한 전략지형도 스펙을 그대로 따를 필요 없이, 훨씬 단순한 별도 스크립트(`powermap_<전투명>.py` 등)로 충분 — 배경색·범례 정도만 기존 스타일과 통일하면 된다. 우선순위 1-보완-B(직접 이미지 URL 임베드)가 성공하면 이 우선순위 2 다이어그램은 생략 가능하다 — 실제 지도를 이미 확보했다면 굳이 생성 다이어그램을 추가로 만들 필요는 없다.

`notion-create-pages`로 페이지를 만들면 그 페이지의 `<page url="...">` 블록이 **부모 페이지 맨 끝에 자동으로 붙는다.** 이걸 원하는 토글 안으로 옮기려면, `update_content` 한 번의 호출 안에 **content_updates 두 개**를 같이 넣어야 한다:
1. 맨 끝에 자동으로 붙은 `<page url="...">...</page>` 를 제거하는 업데이트
2. 원하는 토글 안(사용자 메모 문단 바로 다음, 같은 들여쓰기)에 **똑같은** `<page url="...">...</page>` 태그를 다시 삽입하는 업데이트

그냥 새 위치에 `<page url="...">`를 새로 쓰기만 하면 "이미 존재하는 자식 페이지를 삭제하려는 것이냐"는 validation_error가 난다. (Notion API가 이 두 changes를 한 번에 하라고 에러 메시지에서 직접 알려준다.)
